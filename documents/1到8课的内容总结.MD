## 第 1 课：Claude Code 简介 → Uki 出生证明

- 关键类比：Claude 网页版是顾问（给建议，你动手），Claude Code 是搭档（直接帮你改）

### 🔧 Uki 实现
- 创建 `README.md`，定义 Uki 的身份和能力
- 项目名：UkiAgent，Agent 名：Uki
- 定位：多变的日常助手，通过加载不同 Skill 切换不同身份

## 第 4 课：如何工作 → 代理循环

### 📖 学习内容
- Claude Code 的核心：代理循环（Agent Loop）
- 流程：Prompt → 收集上下文 → 执行动作 → 验证结果 → （循环）
- 两类驱动：模型（大脑）负责思考，工具（手脚）负责执行

### 🔧 Uki 实现
- 重写 `agent.py`：新增 `run()` 方法，实现"思考→行动→观察"的循环
- 新建 `tools.py`：三个工具（list_files, read_file, write_file）
- 最大 10 轮，LLM 决定要不要用工具

### 💬 关键讨论：收集-执行-验证模型
- 用户提出：Agent 的流程是不是「收集→执行→验证」的循环？
- 解答：对。Uki 当前每一步都有收集和执行，但最后的验证环节较薄弱（LLM 给出最终回复后没做外部验证）。后续课程会补（检查点、权限控制、Git 工作流）。

### 💬 关键讨论：Agent 有没有优化用户输入再发给 LLM
- 当前没有。Uki 把用户原话直接塞给 LLM，不做 prompt 增强。
- Claude Code 做了很多上下文增强（附上相关文件、Git 状态等），Uki 还没到这个阶段。
- 这个能力被列为补充课程 S1（Prompt 自动增强）。

### 🔧 Uki 实现
- 扩展 tools.py：新增 `search_code` 工具
  - 文件名搜索（glob 匹配，如 `*.py`）
  - 内容搜索（正则匹配）
  - 自动忽略 .git、__pycache__ 等目录
  - 结果上限 30 条

### 💬 关键讨论：什么该内置，什么该外挂
- 内置标准：做任何任务都大概率要用到的（读文件、搜索）
- 外挂标准：只在特定场景需要的（读 PDF、操作数据库、发邮件）
- 执行命令该内置但排在权限配置之后（安全原因）
- 网络搜索该外挂（适合作为 MCP 演示案例）

## 第 7 课：CLAUDE.md → 规则文件系统

### 📖 学习内容
- Claude Code 的 CLAUDE.md：项目级配置文件，AI 每次对话自动读取
- 两条线的区别：CLAUDE.md（人手写规则）vs MEMORY.md（AI 自动总结偏好）

### 🔧 Uki 实现
- 新建 `UKI.md`：项目规则文件，包含沟通风格、工作习惯、技术约定等
- 修改 `agent.py`：启动时自动读取 UKI.md，注入 system prompt
- `/clear` 命令接入 agent 实例，真正清除对话历史

### 💬 关键讨论：System Prompt 和 UKI.md 的关系
- 初始实现有功能重叠：行为指引（"优先用工具""每次只调一个"）既在 System Prompt 里写了一份，在 UKI.md 里也写了一份
- 清理后：System Prompt 只保留身份定义（"你是 Uki，一个温暖、多变的日常助手"），所有行为指引移到 UKI.md
- UKI.md 相当于 LoRA 的概念——不改内核，改行为配置
- 身份定义留在 System Prompt 里是为了防止 UKI.md 被误删时 Uki 不至于失去名字

### 💬 关键讨论：跨轮次对话历史（上下文）
- 用户发现：第七课的代码中，UKI.md 被传递了，但对话历史没有被传递
- Uki 每次 run() 都从零开始构建 messages，不记得之前说了什么
- 补充实现：添加 `conversation_history` 列表，每次 run() 把历史拼入 messages
- 对话历史输入的原样发送给 LLM，不做增强

## 第 8 课：上下文管理

### 📖 学习内容
- LLM 的上下文窗口是有限的，Agent 必须自己管理好自己的"记忆"
- 三层管理机制：LLM 自动总结（主动瘦身）、工具返回值截断（源头控量）、自动压缩（被动兜底）
- 上下文窗口大小不是"越大越好"：大窗口有注意力衰减、成本高、延迟大三个代价
- 窗口由模型决定：DeepSeek V4 是 1M，GPT-4o 是 128K，Claude 是 200K

### 🔧 Uki 实现

**1. 上下文窗口智能适配**
- `config.py`：`MODEL_WINDOW_CONFIG` 字典，根据模型名自动匹配 window/max/trim/summary
- 四级降级保障：.env 手动覆盖 > OpenRouter API 缓存 > 代码字典 > 保守默认（16K）
- `.env` 支持手动覆盖：`UKI_CONTEXT_WINDOW/MAX/TRIM/SUMMARY_THRESHOLD`
- 新增模型只需在字典加一行或 .env 填三行，不用改核心代码

**2. OpenRouter API 自动获取**
- 新建 `uki/model_fetcher.py`：启动时拉取 https://openrouter.ai/api/v1/models
- 缓存 7 天到 `uki_model_cache.json`
- 自动匹配 500+ 模型的 `context_length`，新模型无需手动配置
- API 挂了降级到缓存和代码字典

**3. LLM 自动总结**
- 在 `run()` 开始前检查 `conversation_history` 的 token 数
- 阈值差异化：小窗口走 30%（如 16K → 4.8K），大窗口 8K 封顶
- 调 LLM 做语义压缩，保留核心需求/决策/约束/未解决问题，丢弃迭代细节
- 摘要限制 800 max_tokens（约 500 字）
- `has_recent_summary` 标记防止连续两轮重复总结

**4. 自动压缩**
- 在 `run()` 循环内检查总 messages 的 token 数
- 超过 max（80% 窗口）触发，裁剪到 trim（60% 窗口）以下
- 保留头（system prompt + 历史 + 用户输入）和尾（最近几轮工具调用），砍中间
- 只在极端情况下触发（如同一轮循环翻了过多页文件）

**5. 工具返回值统一截断**
- `execute_tool()` 最后统一截断：所有返回值上限 4000 字符
- 不同工具不同的提示（read_file 提示翻页，search_code 提示缩小范围）
- LLM 看到截断提示，需要时可以继续翻页

### 💬 关键讨论

**Q: 为什么 LLM 自动总结用绝对值而不是百分比？**
> 总结的目的是"别让注意力衰减影响思考"，衰减发生在绝对 token 位置（第 8000 个 token 注意力已经显著下降），和窗口还剩多少空间无关。百分比适用于压缩（防爆窗口），绝对值适用于总结（防遗忘）。

**Q: 为什么保留头尾而不是全保留最新的？**
> 头提供目标（"帮我写购物车"），尾提供当前状态（"现在加优惠券"），中间是迭代过程（改了三版 UI），对下一步没用。目标和当前位置才是关键。

**Q: 三层防线的关系是什么？**
> 总结管跨轮次的对话历史（主动瘦身），截断管单次工具返回值的大小（源头控量），压缩管循环内临时堆积的总量（被动兜底）。三个守的是不同位置，互不替代。

**Q: 总结和压缩什么关系？压缩会不会砍掉已经总结过的历史？**
> 不会重叠。总结处理 `conversation_history`（跨 run 的），压缩处理单次 run 循环内的工具调用消息。两者是完全不同的数据。

**Q: 工具返回值分页了为什么还会触发压缩？**
> 分页只控单次大小，不控总量。LLM 连翻四页，四页全堆在 messages 里。截断管"每次加多少"，压缩管"总量超了没"。

**Q: 如果真的对话很长，200 字总结够吗？**
> 初始搭配不合理：8K 阈值配 200 字摘要。调整后：8K 阈值保持不变，max_tokens 提到 800（约 500 字），5:1 压缩比能保住关键信息。真不够的场景走分层摘要（记忆系统）。

## 第 1-8 课 Uki 当前能力全景

### 项目结构
```
UkiAgent/
├── UKI.md              ← 项目规则（Uki 每次启动自动读取）
├── README.md           ← 项目说明 + 开发日志
├── requirements.txt    ← Python 依赖
├── .env.example        ← API 配置模板 + 上下文窗口覆盖
├── .gitignore
├── main.py             ← 入口文件
├── uki_model_cache.json ← OpenRouter 模型缓存（7天自动更新）
├── documents/
│   └── study_outline.md ← 完整学习大纲
└── uki/
    ├── __init__.py
    ├── config.py       ← 配置管理 + 模型窗口智能适配
    ├── agent.py        ← 代理循环 + 跨轮次历史 + LLM 总结 + 自动压缩
    ├── tools.py        ← 4 个工具 + 统一截断
    ├── commands.py     ← 7 个命令 + 可扩展注册表
    └── model_fetcher.py ← OpenRouter API 获取
```

### 核心能力
- 代理循环（思考→行动→观察，最多 10 轮）
- 工具集（list_files, read_file, write_file, search_code）
- 跨轮次对话历史（自动带入之前的上下文）
- 上下文管理三层防线（LLM 总结 + 工具截断 + 自动压缩）
- 模型窗口智能适配（OpenRouter API + 代码字典 + .env 覆盖）
- 规则文件系统（UKI.md 自动注入 system prompt）
- 斜杠命令系统（可扩展注册表）

### 消息收集链路
```
发送给 LLM 的内容：
  System Prompt（仅身份定义）
  + UKI.md 内容（所有行为规则）
  + conversation_history（对话历史，超阈值会被总结压缩）
  + 当前用户输入（原样）
  + 循环内上轮工具执行结果（统一截断到 4000 字符）
```
